# GitHub 論文公開リポジトリ構築ガイド：知見と手順

本ドキュメントは、研究論文「Social Hamiltonian and Legal Engineering」のシミュレーションコードを GitHub で公開する過程で得られた技術的ノウハウと、学術的再現性を担保するための手順をまとめたものです。

---

## 1. プロフェッショナルなリポジトリ構成
論文にリンクするリポジトリは、開発時のままでなく、外部の研究者が「どこに何があるか」を即座に理解できる構成にする必要があります。

| フォルダ/ファイル | 役割 |
| :--- | :--- |
| `src/` | メインのプログラム（Pythonなど）。開発時の雑多なスクリプトを除去し、精選する。 |
| `c_impl/` | 高速化のための低レイヤー実装（C/C++など）。 |
| `data/` | 実験結果（JSONデータ）や、論文に使用したグラフ画像（PNG）を格納。 |
| `README.md` | プロジェクトの概要、インストール方法、実行手順を英語で記載。 |
| `requirements.txt` | 実行に必要なライブラリを明記（`pip install -r requirements.txt` で再現可能にする）。 |
| `experiment_log.txt` | 論文の数値データの根拠となる、シミュレーションの実行生ログ。 |

## 2. セキュリティと衛生管理（落とし穴への対策）

### アライアンスとしての「プッシュ保護」
GitHub には、トークンやパスワードを誤ってアップロードするのを防ぐ **Secret Scanning（Push Protection）** があります。
*   **知見**: ドキュメント内やログの中に `ghp_` で始まるアクセストークンが1箇所でも含まれていると、リポジトリ全体のプッシュが拒否されます。
*   **対策**: `findstr` 等で全ファイルを検索し、秘匿情報を完全に除去（サニタイズ）してからプッシュする必要があります。

### `.gitignore` の徹底
`node_modules/`（ライブラリ本体）や作業用の一時ファイル（`.resolved`, `.metadata.json`）はリポジトリに含めてはいけません。
*   クリーンな状態を保つため、最初期に強固な `.gitignore` を設定することが重要です。

## 3. Gitプロトコルが失敗した際の救済策：REST API
環境によって（認証やブランチ保護ルールの影響で）、通常の `git push` が失敗することがあります。
*   **知見**: Gitコマンドが使えない、あるいは強制プッシュ（force-push）がブロックされる場合でも、**GitHub REST API** を利用すれば「ファイル単位」でのアップロードや削除が可能です。
*   **手法**: Node.js 等のスクリプトで API (`PUT /contents/`) を叩くことで、Gitの履歴管理とは別の経路でリポジトリの状態を確定できます。

## 4. 論文（LaTeX）と GitHub の連携
学術論文としての信頼性を高めるため、以下の手順を推奨します。
*   **「Data and Code Availability」セクションの設置**: 参考文献の前に、リポジトリURLを明記した専用セクションを作成する。
*   **ハイパーリンクの検証**: 論文内のURLが正しく、かつリポジトリの「デフォルトブランチ（通常は master または main）」に正しく到達できるかを確認する。

---

## 5. 文字コードと表示の注意
*   **知見**: Windows環境で作成した日本語 `.md` ファイルが `Shift-JIS` で保存されていると、GitHub上で激しく文字化け（Mojibake）します。
*   **対策**: 公開するドキュメントはすべて `UTF-8` で作成するか、内容が不要であれば削除し、ノイズを最小限に抑えます。
